# 需求规格说明

## 1. 产品定位

百码推演是一个面向教学演示与课程项目的兵棋推演 Web 原型。它以排级战术对抗为核心，强调战场态势、单位激活和规则反馈的可视化。

## 2. 用户画像

### 主要用户

- 软件工程课程项目小组成员
- 课堂演示观众
- 对兵棋有兴趣但没有桌游经验的初学者

### 使用场景

- 课堂演示
- 项目中期检查
- 规则讨论与原型验证

## 3. 功能需求

| 编号 | 需求 | 描述 |
| --- | --- | --- |
| FR-01 | 场景加载 | 系统应能加载一个预设战术场景 |
| FR-02 | 对局创建 | 系统应能基于场景创建新对局 |
| FR-03 | 单位激活 | 当前行动方可选择未行动单位开始激活 |
| FR-04 | 单位机动 | 单位可在规则允许范围内移动到目标地块 |
| FR-05 | 射击判定 | 单位可对视线内目标执行射击并得到结果 |
| FR-06 | 状态更新 | 系统应维护压制、消灭、行动点等状态 |
| FR-07 | 回合推进 | 系统应自动推进激活序列与回合 |
| FR-08 | 胜利统计 | 系统应展示目标区控制与伤亡统计 |
| FR-09 | 作战日志 | 系统应记录关键行动并向用户展示 |
| FR-10 | 重开对局 | 用户可快速重置当前场景重新开始 |

## 4. 非功能需求

| 编号 | 需求 | 描述 |
| --- | --- | --- |
| NFR-01 | 易演示 | 主要操作应能通过点击完成 |
| NFR-02 | 可维护 | 规则引擎应与前后端显示逻辑分离 |
| NFR-03 | 可扩展 | 代码结构应支持增加新规则和场景 |
| NFR-04 | 可测试 | 核心规则应支持单元测试 |
| NFR-05 | 响应及时 | 单次命令响应应在课堂演示条件下保持流畅 |

## 5. 典型用例

### UC-01 开始一局新对局

- 参与者：玩家
- 前置条件：系统已启动
- 主流程：
  1. 玩家进入页面
  2. 系统加载默认场景
  3. 玩家点击“重开本局”或切换场景
  4. 系统创建新对局并返回初始状态

### UC-02 激活并移动单位

- 参与者：玩家
- 前置条件：存在当前行动方可用单位
- 主流程：
  1. 玩家点击己方单位
  2. 系统将其设为当前激活单位
  3. 玩家点击相邻目标地块
  4. 系统校验地形、占位与行动点
  5. 系统更新位置与日志

### UC-03 射击敌方单位

- 参与者：玩家
- 前置条件：已选中当前激活单位
- 主流程：
  1. 玩家点击敌方单位
  2. 系统校验射程与视线
  3. 系统执行掷骰与伤害判定
  4. 系统更新目标状态、日志与胜利进度

## 6. MVP 裁剪原则

为保证课程周期内可交付，MVP 只实现最关键的战术闭环，不强求完全复刻原规则。

### MVP 必须完成

- 单场景
- 单局内状态管理
- 机动与射击主循环
- 前端战场展示
- API 通信

### 可延期需求

- 多房间联机
- 场景编辑器
- 战报回放
- AI 敌人
- 完整规则参数化配置

## 7. 验收标准

### 基础验收

- 运行 `npm run dev` 后可进入界面
- 页面能显示地图、单位和日志
- 执行移动与射击后状态有变化

### 课程验收

- 文档齐全
- 架构层次清晰
- 代码能支撑汇报演示
